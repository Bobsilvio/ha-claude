## 4.2.0 â€” Entity discovery: use real HA device_class instead of keyword matching
**Breaking change in entity matching logic â€” eliminates false positives entirely.**

### Problem
Previous approach used keyword/substring matching on entity_ids to find entities (e.g. searching for "battery" by matching "bat" inside entity names). This caused false positives: "bat" matched "sabato", pulling in unrelated consumption entities. Every new device_class would require a new keyword dictionary â€” fragile and unscalable.

### Solution
- `intent.py`: **Two-mode entity discovery:**
  - **Device-class mode** (battery, temperature, humidity, etc.): filters ONLY by the REAL `device_class` attribute from Home Assistant state â€” zero false positives, no substring matching needed
  - **Keyword mode** (fallback): for brands, room names, or custom terms that have no device_class mapping â€” still searches entity_id/friendly_name
- `intent.py`: removed `_keyword_synonyms` dictionary entirely â€” no longer needed since device_class filtering doesn't require synonym expansion
- `intent.py`: expanded `_device_class_aliases` to cover both IT and EN terms (batterie, battery, temperatura, temperature, etc.)
- `tools.py`: `_inject_entity_filter_fallback()` simplified â€” trusts the backend entity list as authoritative, removed all `_dc_keywords` dictionaries and keyword-based re-filtering

### Result
Works for any device_class (battery, temperature, motion, humidity, etc.) without maintaining keyword vocabularies. New device types work automatically.
